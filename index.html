JMM 和 Thread

为解决 CPU 与内存速度的不匹配，加入了高速缓存来作为内存与处理器之间的缓冲，但也引入了一个问题：缓存一致性。每个处理器拥有自己的高速缓存，用共享同一块主内存。当多个处理器的运算任务都涉及同一主内存时，可能导致各自的缓存数据不一致。为解决此问题，在访问缓存时需遵守一些特定的协议。

Java 内存模型

主内存和工作内存

Java 内存模型的主要目标是定义程序中各个变量的访问规则，即虚拟机将变量存储到内存以及从内存读取变量的底层细节。这里所说的变量包括实例字段、静态字段和构成数组对象的元素，不包括局部变量和方法参数，因为它们是线程私有的。

Java 内存模型规定了每个变量存储在主内存中，此外，每条线程还有自己的工作内存。线程对变量的操作必需在工作内存中进行，不能直接读写在主内存的变量。

内存间交互操作

关于主内存和工作内存之间的交互协议，Java 内存模型中定义了八种操作：

  • lock
  • unlock
  • read
  • load
  • use
  • assign
  • store
  • write

volatile

被 volatile 修饰的变量具有两种特性：

  1. 保证此变量对所有线程的可见性。可见性是指当一条线程修改了此变量的值，新值对其它线程来说是立即得知的。虽然 volatile 变量在各个工作内存是一致的，但 Java 里面的运算并非原子操作，因此 volatile 变量的运算在并发过程中一样是不安全的。
  2. 禁止指令重排序。

由于 volatile 只能保证可见性，对于不符合以下两条规则的运算场景，仍需通过加锁来保证原子性。

  • 运算结果并不依赖变量当前值，或者能确保只有单一的线程修改变量的值
  • 变量不需要与其它状态变量共同参与不变约束

虚拟机字节码执行引擎

虚拟机是一个相对于 ”物理机“ 的概念，这两种均有执行代码的能力，其区别是物理机的执行引擎直接建立在处理器、硬件、指令集和操作系统层面上，而虚拟机的执行引擎是自己实现的，因此可以自行制定指令集与执行引擎的结构体系，执行不被硬件直接支持的指令集格式。

运行时栈帧结构

栈帧是用于支持虚拟机进行方法调用和执行的数据结构，是虚拟机运行时数据区的虚拟机栈的栈元素。每个方法从调用到完成的过程都对应着一个栈帧在虚拟机栈从入栈到出栈的过程。

每一个栈帧包括了局部变量表、操作数栈、动态连接、方法返回地址和附加信息。在编译期，栈帧需多大的局部变量表、多深的操作数栈已完全确定，并且写入到方法表的 Code 属性中。

在活动线程中，只有位于栈顶的栈帧才是有效的。

  • 局部变量表

    存放方法参数和方法内部定义的局部变量，以变量槽（Variable Slot）为最小单位，存放一个 boolean, byte, char, short, int, float, reference, returnAddress 类型的数据。一个局部变量定义了但没赋初始值不能使用。局部变量表中的 Slot 是可以复用的。

  • 操作数栈

    方法执行过程中，会有各种指令往操作数栈写入和读取内容。

  • 动态连接

    每个栈帧都包含一个指向运行时常量池中该栈帧所属方法的引用，其目的是为了支持方法调用过程中的动态连接。

  • 方法返回地址

    一个方法开始执行后，有两种方式可以退出这个方法：

    1. 正常完成出口：执行引擎遇到任意一个返回的字节码指令。
    2. 异常完成出口：方法执行过程中遇到了异常，并且这个异常没有在方法体内得到处理，在本方法的异常表中没有搜索到匹配的异常处理器，就会导致方法退出。

    方法退出后，需要返回到方法被调用的位置。正常退出时，调用者的 PC 计数器的值可作为返回地址；异常退出时，返回地址通过异常处理器来确定。

类加载机制

类加载的生命周期：加载，验证，准备，解析，初始化，使用，卸载。其中，验证，准备，解析统称为链接。加载，验证，准备，初始化，卸载的顺序是确定的，但并非按顺序依次执行，而是按顺序依次开始，而后通常是交叉进行。解析阶段在某些情况下可在初始化之后再开始，这是为了支持 Java 运行时绑定的特性。

类加载的时机

虚拟机并没有强制约束何时进行加载过程，但对于初始化过程，虚拟机严格规定了有且只有五种情况必须对类进行初始化：

  • 遇到 new, getstatic, putstatic, invokestatic 指令，若类没有进行过初始化，则需先初始化。最常见的 Java 代码场景：使用 new 关键字实例化对象，读取或设置一个类的静态字段，调用一个类的静态方法的时候
  • 对类进行反射调用的时候，若没初始化，则初始化
  • 当初始化一个类的时候，其父类还没初始化，则需先初始化其父类
  • 虚拟机启动时，用户需指定一个要执行的主类，虚拟机会先初始化这个主类
  • 如果一个 java.lang.invoke.MethodHandle 实例最后的解析结果 REF_getStatic, REF_putStatic, REF_invokeStatic 的方法句柄，并且这个方法句柄所对应的类没有初始化，则先初始化

以上五种场景称为对一个类的主动引用，其他引用类的方式都不会初始化，称为被动引用。例如：

  • 子类引用父类的静态字段，不会导致子类初始化。
  • 常量在编译阶段会存入调用类的常量池中，本质上并没有直接引用到定义常量的类，因此不会触发类的初始化。
  • 通过数组定义来引用类，不会触发类的初始化。如：SuperClass[] sca = new SuperClass[10]

类加载过程

  • 加载

    加载过程需完成三件事：

    1. 通过一个类的全限定名来获取定义此类的二进制字节流
    2. 将这个字节流所代表的静态存储结构转化为方法区的运行时数据结构
    3. 在内存中生成一个代表这个类的 java.lang.Class 对象，作为方法区这个类的各种数据的访问入口

  • 验证

    确保 Class 文件的字节流中包含的信息符合当前虚拟机的要求，并且不会危害虚拟机自身的安全。Java 语言对于一些不安全的行为，如访问数组边界以外的数据，将拒绝编译。但 Class 文件并非仅能来自 Java 源码，它可以使用任何途径产生。因此，需要验证这一过程。

  • 准备

    正式为类变量分配内存并设置类变量的初始值。这里并不包括实例变量；其次，这里所说的初始值一般是指零值。如 public static int value = 123; ，那么准备阶段过后 value 的值应该是 0。将其赋值为 123 是在初始化阶段完成。也存在特殊情况：如果类字段的字段属性表中存在 ConstantValue 属性，那么在准备阶段就会初始化为所指定的值，如 public static final int value = 123。

类文件结构

字节码是 Java 实现 “ Write Once, Run Anywhere ” 的基石。字节码存储于程序编译后产生的 Class 文件中。Class 文件是一组以八个字节为基础单位的二进制流。Class 文件格式采用两种数据类型来存储数据：

  • 无符号数

    以 u1，u2，u4，u8 分别代表 1，2，4，8 个字节的无符号数，无符号数可用来描述数字，索引引用，数量值或者按 UTF-8 编码构成的字符串值。

  • 表

    由多个无符号数或者其他表作为数据项构成的复合数据类型，习惯以“_info”结尾。

Class 文件本质上就是一张表。下面让我们来看一下 Class 文件具体的数据项内容。

  1. 魔数与Class文件的版本

    每个类文件头四个字节称为魔数，作用是确定这个文件是否能被虚拟机接受。

    类文件的魔数：0xCAFEBABE。

    紧接着的四个字节存储的是类文件的版本号。第 5，6 字节为次版本号，第 7，8 字节为主版本号。高版本JDK向下兼容以前版本的类文件。

  2. 常量池

    紧接着版本号的是常量池入口，它是类文件中的资源仓库。

    在常量池入口放置着 u2 类型的数据，代表常量池的容量计数值，确定常量池中的常量数量，容量计数从 1 开始而非 0。

    常量池主要存放：字面量和符号引用。

    字面量比较接近 Java 语言层面的常量概念，如文本字符串，声明为 final 的常量。

    符号引用包括了三类常量：

    • 类和接口的全限定名
    • 字段的名称和描述符
    • 方法的名称和描述符。

  3. 访问标志

    紧接着两个字节代表访问标志，用于识别一些类或接口层次的访问信息。包括：是类还是接口，是否为public，是否为abstract，是否声明为final等等。

  4. 类索引，父类索引，接口索引集合

    类索引和父类索引均是一个 u2 类型的数据，而接口索引是一组 u2 类型的集合，因为 Java 支持单继承多实现。它们用于确定这个类的继承关系。类索引确定这个类的全限定名，父类索引确定这个类的父类的全限定名，接口索引集合描述这个类实现了哪些接口。

  5. 字段表集合

    用于描述接口或者类中声明的变量，不包括方法内部声明的局部变量。描述的信息包括：字段的作用域（public, private 等）、是实例变量还是类变量（static）、是否可变（final）、并发可见性（volatile）、可否被序列化（transient）、字段数据类型、字段名称。

  6. 方法表集合

    类似字段表。包括访问标志，名称索引，描述符索引，属性表集合。方法中的代码存放在方法属性表集合中一个名为”Code“的属性里。

  7. 属性表集合

    • Code 属性

      出现在方法表的属性集合中，接口或抽象类中不存在此属性。主要包括了属性名称、属性值长度、操作数栈最大深度、局部变量表的存储空间、字节码指令和长度。字节码指令长度虽是一个 u4 的值，但实际只使用了 u2 的长度。因此，如果方法过长，生成的字节码指令过长，将无法编译。在字节码指令后是异常处理表集合（非必需存在）。

    • Exceptions 属性

      列举方法中可能抛出的受查异常，即方法描述时 throws 关键字列举的异常。

    • LineNumberTable 属性

      描述 Java 源码行号与字节码行号之间的对应关系。非必需。作用之一是抛出异常时显示出错的行号。

    • SourceFile 属性

      记录生成这个 Class 文件的源码文件名称。

    • InnerClasses 属性

      记录内部类与宿主类的关联。

    • Deprecated 和 Synthetic 属性

      Deprecated 用于某个类、字段或方法，表明已不推荐使用。

      Synthetic 代表此字段或方法不是由 Java 源码直接产生，而是编译期自行添加的。

垃圾收集器与内存分配策略

对象已死吗

引用计数法

给对象添加一个引用计数器，当有一个地方引用它时，计数器值加 1，引用失效时，计数器减 1，当计数器为 0 的对象就是不再被使用的。目前主流的 Java 虚拟机不使用此种方法，原因是它难以解决对象之间相互循环引用的问题。例如，对象 A 和对象 B 都有字段 instance，令 A.instance = B 以及 B.instance = A ，此时它们互相引用着对方，除此之外没有其它引用，实际上已经不可能被访问到，但它们的计数都不为 0，无法被 GC 回收。

可达性分析法

从GC Roots 到对象不可达时，则证明此对象是不可用的。GC Roots 的对象包括：

  • 虚拟机栈中引用的对象
  • 方法区中类静态属性引用的对象
  • 方法区中常量引用的对象
  • 本地方法栈中JNI 引用的对象。

再谈引用

Java 中引用分为四种：

  • 强引用

    类似 Object obj = new Object() 之类的引用，只要强引用还在，GC 则不会对其进行回收。

  • 软引用

    用来描述一些有用但非必须的对象。在发生内存溢出之前，GC 会回收此类对象。Java 中提供了 SoftReference 类来实现软引用。

  • 弱引用

    用来描述非必需的对象。强度必软引用更弱，被弱引用关联的对象只能生存到下一次 GC 之前，GC 工作时，无论内存是否足够，都会回收。对应的类为 WeakReference。

  • 虚引用

    最弱的一种引用关系。无法通过虚引用取得对象实例。为一个对象设置虚引用的唯一目的是能在该对象被回收时收到一个系统通知。对应的类为 PhantomReference。

回收方法区

主要回收两部分内容：废弃常量和无用的类。无用的类需满足下面三个条件：

  • 该类的所有实例已被回收
  • 加载该类的 ClassLoader 已被回收
  • 该类对应的 java.lang.Class 对象没有在其它地方被引用，无法通过反射访问该类的方法

垃圾收集算法

  • 标记-清除算法

    标记所要回收的对象，标记完成后统一回收。缺点：标记和清除两个过程效率不高；标记清除后产生大量不连续的内存碎片。

  • 复制算法

    将可用内存分为大小相等的两块，每次只使用其中一块。当一块内存用完了，将还存活的对象复制到另一块上面，再把已使用过的内存空间清理掉。实现简单，高效，代价是将内存缩小为原来的一半。在新生代中，对象 98% 是“朝生夕死”的，因此不必按照 1：1 来划分内存空间，而是将内存分为一块较大的 Eden 空间和两块较小的 Survivor 空间，每次使用 Eden 和其中一块Survivor。回收时，将还存活的对象复制到另一块 Survivor 空间上，清理掉另外两块空间。Eden和Survivor默认大小比例是 8：1。若 Survivor 空间不够，将依赖其它内存进行分配担保。

  • 标记-整理算法

    老年代中对象存活率一般较高，不使用复制算法。与标记-清除算法类似，不同的是后续步骤不是直接回收对象，而是让所有存活的对象向另一端移动，然后直接清理掉端边界外的内存。

  • 分代收集算法

    把Java堆分为新生代和老年代，为其选择不同的算法。

HotSpot 的算法实现

枚举根结点

可达性分析过程中必需确保一致性，即分析过程中对象的引用关系不可变，这就要求 GC 进行时必需停顿所有 Java 执行线程。

另外，可达性分析过程中若要一个不漏地检查完执行上下文和全局的引用位置，需耗费很多时间。因此，在 HotSpot 中，使用一组称为 OopMap 的数据结构来直接得知哪些地方存放着对象的引用，而不需逐个检查。

安全点

若为每条指令生成 OopMap 将会耗费大量空间。故实际中只在特定位置记录了这些信息，这些位置称为安全点。程序执行时只有到达安全点才能暂停。为了在 GC 发生时所有线程跑到最近的安全点才停下来，常采用主动式中断的方式：当 GC 需中断时，不直接对线程进行操作，仅设置一个标志，让个线程轮询此标志，发现中断标志为真时自己就中断挂起，轮询标志与安全点是重合的。

安全区域

程序不执行时，例如线程处于 Sleep 或 Blocked 状态，此时线程无法响应中断请求。为解决此问题，采用安全区域。安全区域是指在一段代码片段内，引用关系不会发生改变，在此区域中任何地方进行 GC 是安全的。

线程执行到 Safe Region 时，标志自己进入了此区域，在这段时间内，若 JVM 要进行 GC，就不用管标志为安全区域的线程了。当线程要离开 Safe Region 时，需先检查是否完成了根结点枚举，若完成则线程继续执行，否则需等待知道收到可安全离开安全区域的信号为止。

垃圾收集器

首先来了解下并行和并发的概念：

并行：多条垃圾收集线程并行执行，而用户线程处于等待状态。

并发：用户线程与垃圾收集器线程同时执行，但不一定是并行的，可能是交替执行。

  • Serial 收集器

    单线程收集器，使用一个 CPU 或一条收集线程收集，收集时必需暂停所有线程。优点：简单而高效（与其它收集器的单线程相比）。Client 模式下的虚拟机的选择。

  • Parnew 收集器

    Serial 收集器的多线程版本。使用多条线程进行垃圾收集。在单 CPU 环境下性能不比 Serial 好。Server 模式下的虚拟机中首选的新生代收集器，其中一个原因是目前除了 Serial 收集器，只有它能与 CMS 收集器配合工作。

  • Paralel Scavenge 收集器

    并行多线程的新生代收集器，采用复制算法。与上面两个收集器不同，其关注的不是缩短停顿时间，而是吞吐量。所谓吞吐量，举个例子，虚拟机运行了 100 分钟，其中 GC 进行了 1 分钟，吞吐量就是 99%。

  • Serial Old 收集器

    Serial 收集器的老年代版本，使用标记-整理算法。

  • Parallel Old 收集器

    Paralel Scavenge 的老年代版本，使用多线程和标记-整理算法。

  • Cms 收集器

    采用标记-清除算法。收集主要包括四个步骤：

    1. 初始标记：仅标记一下 GC Roots 能直接关联的对象，速度快
    2. 并发标记：进行 GC Roots Tracing 的过程
    3. 重新标记：修正并发标记过程中因用户程序运行而导致标记产生变动的那一部分对象
    4. 并发清除

    因耗时较长的并发标记和并发清除过程可以与用户线程一起工作，故停顿时间较短。

    其缺点主要有：

    1. 对 CPU 资源敏感。在并发阶段，虽不会导致用户线程停顿，但会占用线程资源而使程序变慢。
    2. 无法处理浮动垃圾。并发清除阶段用户线程仍在运行，会产生垃圾。因此，在清除过程中需预留一定内存空间供用户线程使用。若预留的内存不足，虚拟机将临时启用 Serial Old 收集器重新进行收集，导致停顿时间变长。
    3. 标记-清除算法会产生大量内存碎片。

  • G1 收集器

    主要包括以下四个步骤：

    1. 初始标记
    2. 并发标记
    3. 最终标记
    4. 筛选回收

内存分配

  • 对象优先在 Eden 区分配

  • 大对象直接进入老年代

  • 长期存活的对象将进入老年代

    若对象出生在 Eden ，在进行一次 Minor GC 后仍存活并能被 Survivor 容纳，则将对象的年龄加一，当年龄达到指定大小时将晋升到老年代中。

  • 动态对象年龄判定

    如果在 Survivor 空间中相同年龄的对象的大小大于总空间的一半，则年龄大于等于这些年龄的对象将进入老年代，而无需等到指定年龄。

  • 空间分配担保

    发生 Minor GC 前，虚拟机先检查老年代的最大可用的连续空间是否大于新生代所有对象的总空间，若条件成立，则可确保 Minor GC 是安全的。若不成立，则查看 HandlePromotionFailure 是否允许担保失败，如果允许，则将检查老年代最大可用连续内存空间是否大于历次晋升到老年代的对象的平均大小，若大于，将尝试进行一次 Minor GC，尽管有风险；若小于，或 HandlePromotionFailure 不允许担保失败，则进行一次 Full GC。

Java内存区域与内存溢出

运行时数据区域

Java 虚拟机在执行 Java 程序过程中会把它所管理的内存划分为若干个不同的数据区域。

  • 程序计数器

    可看作当前线程所执行的字节码的行号指示器，通过改变计数器的值来选取下一条需要执行的字节码指令。每个线程有一个独立的计数器，各线程互不影响。此内存区域是虚拟机唯一一个没有规定OOM的区域。

  • Java虚拟机栈

    线程私有。描述 Java 方法执行的内存模型。方法执行的同时会创建一个栈帧用于存储局部变量表，操作数栈，动态链接，方法出口等信息。局部变量表存放了编译期可知的各种基本数据类型，对象引用类型和返回地址类型，在方法运行期不会改变局部变量表的大小。若线程请求的栈深度大于虚拟机所允许的深度，将抛出 StackOverflowError 异常，若虚拟机栈可动态扩展，且扩展时无法申请到足够内存，就会抛出 OOM。

  • 本地方法栈

    与虚拟机栈类似。区别为虚拟机栈为虚拟机执行Java 方法服务，而本地方法栈为虚拟机使用到的 Native 方法服务。

  • Java 堆

    存放对象实例。线程共享。

  • 方法区

    线程共享。用于存储已被虚拟机加载的类信息，常量，静态变量，即时编译期编译后的代码等数据。

  • 运行时常量池

    方法区的一部分。用于存放编译期生成的各种字面量和符号引用。

Java中的内存泄漏，是指对象无用但仍被持有，此时对象无法被 GC 回收，占据着内存。

常见的内存泄漏

集合类泄漏

集合类中的元素被添加进去后没有删除，此时如果仅仅释放了元素的引用，则集合类仍持有元素对象的引用，便产生了内存泄漏。
解决办法：移除元素，或者直接将集合类赋为 null 。
